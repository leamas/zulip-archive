It works also with testplugin, but the steps are more convoluted involving a "fake" upload to create the tarball

TwoCan (May 04 2026 at 19:11):

Or for us pesky Windows developers...
mkdir build
cd build
cmake .. -A Win32
cmake --build . --config MinSizeRel|RelWithDebInfo|Release|Debug --target tarball
and then just use Plugin Manager to import the plugin.

As Alec suggested, once it has been initially installed, for subsequent builds just copy the newer plugin .dll (or for other platforms .so) into the appropriate directory.

Quinton Hoole (May 05 2026 at 00:16):

@seandepagnier @sean d'epagnier I personally try to cover as much of the code as possible with automated tests. Then just “make test” runs them all. You can see some such tests, coverage reports etc in the weather routing plugin. Details at the bottom of the reader page.

https://github.com/rgleason/weather_routing_pi/blob/master/README.md

It’s obviously not possible to automatically test all the GUI functionality, but it’s pretty straightforward to automatically test non-GUI code. That at least cuts down on how many times you need to restart OpenCPN and do manual testing.

Incidentally, Claude Code AI seems to do a reasonably decent job of writing unit tests for you automatically, if you’re into that sort of thing.
