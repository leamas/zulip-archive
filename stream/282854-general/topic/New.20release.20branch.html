I plan to create a new branch called "v5.8.x" wherein we can confine and manage any backport activity

Yes, we need that. May I suggest that we apply some simple (but actually quite strict) rules for this branch?

  1. All new commits goes into master. v5.8.x only contains cherry-picked commits from master.
  2. All commits in v5.8.x should have a bug reference.

These are traditional rules for release branches. The first rule makes sure that v5.8.x does not diverge from master with new funcitonality, we don't want that. The second makes sure that all commits in v5.8.x fixes a documented problem

THoughts?
