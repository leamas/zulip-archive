Now NavMsg is defined in comm_navmsg.h like:

class NavMsg {
public:
  NavMsg() = delete;

  virtual std::string key() const = 0;

  virtual std::string to_string() const {
    return NavAddr::BusToString(bus) + " " + key();
  }

  const NavAddr::Bus bus;

protected:
  NavMsg(const NavAddr::Bus& _bus) : bus(_bus){};
};

Obviously, here is no such string. OTOH, Nmea0183Msg is :

class Nmea0183Msg : public NavMsg {
public:
  ...
  std::string id;      /**<  For example 'GPGGA'  */

};

Dave Register (Aug 15 2022 at 03:29):

Just to get more eyes on this, consider:
There are basically four items in global data that we must prioritize.

  1. Position (lat/lon)
  2. Velocity (sog/cog)
  3. True Heading (Hdt)
  4. Mag variation (Var).
    The complication is that N0183 and n2k messages may provide updates on one or more of these items. Only RMC provides them all, usually. VTG provides only velocity, GLL only position, etc. Also, N0183 allows the possibility of "null" fields in the message. So it it not sufficient to make priority decision based on the message alone.
    My solution, hinted by Hakan, is to prioritize by category separately, based on the four items mentioned above. This of course adds another dimension to the problem, leading to some not-so-pretty code. And then the scenario of similar N0183 messages from separate driver streams. Another dimension. But that is what we have to deal with.
