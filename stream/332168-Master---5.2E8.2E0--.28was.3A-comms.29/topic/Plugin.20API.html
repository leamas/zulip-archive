Dave Register (Jul 31 2022 at 19:38):

I suggest wrapper functions, with a defined set of possible function arguments in the API header.
For plugins that listen only (i.e. most of them), we have API functions like:
int listen_N0183( std::string identifier);
int listen_N2K( uint32 pgn);
etc.
To notify a plugin, there are choices:

  1. we adapt our notify() methods to call a handler in plugin_manager that wraps up the data, and makes a defined API call to the plugin. Plugins are not receiving wxEvents in this idea.
  2. Or define and exposes the wxEvent structure contained in the core.
    I prefer something like (1), and I'm sure the plugin authors will too. They are already processing a number of "direct call" functions to get NMEA0183 data in our now-legacy model, for example.

Plugins that need to write to a specific N2K driver: More thought required, including some registry enumeration/GETCAPS notion.

As I see it, the plugin interface can be done in one of two quite different ways.

  1. Try (very hard) to hide the listener logic in pluginmanager.cpp. Present a very simple API to plugin that allows plugins to register a request for specific comm messages. Call methods in plugin object directly as those messages are received by pluginmanager. No comm event handling in plugin. This will be seen as a simple extension of current plugin logic.

  1. Refactor comm headers, presenting a simpler subset to the plugin API. Teach plugin authors about observable life cycle, events, Bind() lambda, etc. Better logic? Arguably, yes. More complex? For sure.

Dave Register (Aug 03 2022 at 14:00):

This discussion applies to N0183 only, for legacy support.
Taking a test case for example. The extremely popular VDR plugin. This plugin simply wants to receive (by direct method call) the entire N0183 data stream. What I propose is that a listener for "ALL N0183" shall be instantiated in pluginmanager. On event callback, pluginmanager sends the N0183 string to all requesting plugins. We cannot drop this capability.
Same situation for the multiplexer. If a N0183 comm connection is marked as "output", users expect to see "ALL n0183", with filters applied as per connection GUI.
We could do this the hard way, and pre-define a list of N0183 sentences that OCPN, or plugins now and future, might possibly have an interest in. And then listen on them all. But that seems not great. Bound to miss something...
Related topic: We generally do not care about who the NMEA0183 "talker" is. That is, O treats "GPRMC" the same as "IIRMC". So the observable will need to handle this case, since we have no a-priori information on who a "talker" will be. Thus, the wildcard "??RMC"

So I see three classes of plugin support, all operable in parallel.

  1. Pure legacy, as we do today, and many users depend on. Handles N0183 and sK streams only.
  2. Extended Legacy. My proposal, if I can make it work. I'll drop this soon, if it becomes a can of worms.
  3. Full-on "advanced", with fully exposed observable logic. This will be needed for any new plugin that deals with n2k directly.
