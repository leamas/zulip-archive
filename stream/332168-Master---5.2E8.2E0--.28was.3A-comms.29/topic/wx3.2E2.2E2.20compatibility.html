Kees Verruijt (Apr 08 2023 at 21:11):

@Pavel Kalian was right: the problem is EGL vs GLX. The default compilation path for wx is to use EGL, as that is "the future of OpenGL" (https://wiki.archlinux.org/title/User:CodingKoopa/Direct_Rendering_Infrastructure and https://www.reddit.com/r/opengl/comments/11q0oz/what_is_the_difference_between_eglglxogl_and_gles/ ).

Debian has chosen to go with GLX instead of EGL for wx 3.2.2: https://bugs.debian.org/cgi-bin/bugreport.cgi?bug=1024147

So we are now in a bad place whatever way we go:

  • the wx 3.2.2 libraries that come with Debian Sid are compiled with --disable-glcanvasegl. Indeed if I compile the plugin on Sid I get a dependency on _ZNK13wxGLCanvasX1115IsShownOnScreenEv@WXU_3.2. Not sure where @Dave Register got his wx 3.2.2 for Ubuntu 22.04, but according to the changelog for Ubuntu they just pull Debian: http://changelogs.ubuntu.com/changelogs/pool/universe/w/wxwidgets3.2/wxwidgets3.2_3.2.2+dfsg-2build1/changelog
  • the wx 3.2.1 libraries compiled by Alec that most plugins are compiled agains _do_ use EGL (because he probably used the default compiler flags...)

So ... if we want to stand a chance of supporting O and radar_pi in the O 5.8 timeframe on Debian/Ubuntu, we need to recompile the plugin to use --disable-glcanvasegl (and keep the O production PPA as it is.)

In other words, in my opinion, the O "production" PPA is correct and we need to fix the plugins to use --disable-glcanvasegl as well.

Kees Verruijt (Apr 11 2023 at 13:55):

Hey, I didn't invent this :wink: just copied it from ci/circleci-build-debian.sh which is what is used for x86_64.

Setting BUILD_WX32 results in a -DOCPN_WX_ABI=wx32 passed to cmake, which results in set(_wx_abi ".${OCPN_WX_ABI}") being set to wx32, and this in turns results in pkg_displayname and pkg_tarname gaining .wx32. It's also messing slightly with flatpak package name.

The result is still messy, as the list of my .xml files shows:

Radar-v5.5.0-beta2-debian-11.xml
Radar-v5.5.0-beta2-debian-12.xml

Radar-v5.5.0-beta2-debian-A32-12.xml
Radar-v5.5.0-beta2-debian-A64-11.xml
Radar-v5.5.0-beta2-debian-A64-12.xml
Radar-v5.5.0-beta2-debian-wx32-A64-11.xml
Radar-v5.5.0-beta2-debian-wx32.wx32-11.xml
Radar-v5.5.0-beta2-debian-wx32.wx32-A32-11.xml
Radar-v5.5.0-beta2-flatpak-22.08.xml
Radar-v5.5.0-beta2-flatpak-A64-22.08.xml
Radar-v5.5.0-beta2-msvc-wx32-10.xml

Note how the A32 and x86_64 (no arch in name) both have wx32 in the package name twice.

Could it be that this OCPN_WX_ABI_OPT was used more in the past? At the moment it's only used in ./cmake/Metadata.cmake to fiddle with package and tar names. I think the better approach is to use only OCPN_TARGET_TUPLE. As a fail-safe the CMake files could verify that a tuple including -wx32 is indeed using wx3.2.

What do you prefer? Fixing this yourself in SD or me preparing a change in radar_pi? A simple fix would be to remove the double .wx32 name from the filenames, but I would prefer to make a better fix so we don't have this confusion/double code at all.
